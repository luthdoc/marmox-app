# Story 5.1: Schema owner_phone e Funções de Escrita no Tenant

**Epic:** 5 — Onboarding Agent
**Status:** completed

## User Story

As a developer,
I want the tenants table to have an `owner_phone` column and write functions in `db/tenants.py`,
so that the onboarding agent can know who to talk to and save the collected configuration.

## Acceptance Criteria

1. A migration `005_owner_phone.sql` adiciona `owner_phone TEXT` à tabela `tenants` com `ADD COLUMN IF NOT EXISTS` (idempotente).
2. `db/tenants.py` exporta `update_tenant_config(tenant_id: str, config: dict) -> None` que persiste `name`, `services`, `regions`, `business_hours` e `welcome_message` no tenant, chamando `set_tenant_context(tenant_id)` antes de qualquer query (NFR3).
3. `update_tenant_config` ignora silenciosamente chaves não reconhecidas do `config` — apenas os campos listados no AC2 são escritos.
4. `db/tenants.py` exporta `complete_onboarding(tenant_id: str) -> None` que atualiza `status = 'active'` no tenant, chamando `set_tenant_context(tenant_id)` antes da query (NFR3).
5. `db/tenants.py` exporta `get_owner_phone(tenant_id: str) -> str | None` que lê `owner_phone` do tenant chamando `set_tenant_context(tenant_id)` antes da query (NFR3) — consolida a lógica duplicada de `notification_service._get_owner_phone`.
6. `notification_service._get_owner_phone` é removida e substituída por chamada a `db.tenants.get_owner_phone`.
7. Testes unitários em `backend/tests/test_tenants_db.py` verificam `update_tenant_config`, `complete_onboarding` e `get_owner_phone` com mocks do cliente Supabase.
8. Todos os testes CI existentes continuam passando após a refatoração de `notification_service`.

## Technical Notes

> `notification_service.py` já possui uma função privada `_get_owner_phone` que faz exatamente o que o AC5 descreve. Mover para `db/tenants.py` elimina a duplicação antecipada pela Story 5.3.
>
> A migration deve ser executada no Supabase SQL Editor. O arquivo fica em `backend/db/migrations/005_owner_phone.sql`.
>
> `update_tenant_config` deve usar `.update(filtered_data).eq("id", tenant_id).execute()` na tabela `tenants`. Não criar nova migration para novos campos — todos já existem desde `003_tenant_context.sql`.
