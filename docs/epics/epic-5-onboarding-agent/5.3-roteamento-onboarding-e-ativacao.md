# Story 5.3: Roteamento de Onboarding e Ativação do Tenant

**Epic:** 5 — Onboarding Agent
**Status:** completed

## User Story

As a marble company owner,
I want messages I send during setup to be answered by the onboarding agent, and for the system to automatically activate my account when I finish providing all required information,
so that the agent starts serving my customers without any manual intervention on the platform's side.

## Acceptance Criteria

1. `webhook_service._handle_inbound_message` adiciona um branch `elif msg.tenant_status == "onboarding":` que dispara `_dispatch_onboarding_agent` via `asyncio.create_task`, sem alterar o branch `active` existente.
2. `_dispatch_onboarding_agent(tenant_id, owner_phone, text)` carrega o histórico da conversa via `load_conversation_history(tenant_id, owner_phone)`, chama `process_onboarding_message` e envia a resposta via `send_message`.
3. A mensagem de resposta enviada ao dono não contém o bloco `[DADOS_EMPRESA]` — ele é removido antes do `send_message` (mesmo padrão de `[DADOS_LEAD]` no agente principal).
4. Se `parse_empresa_block` retornar `onboarding_complete == True`, `_dispatch_onboarding_agent` chama `update_tenant_config(tenant_id, dados)` e em seguida `complete_onboarding(tenant_id)`, mudando `status` para `active`. Ambas as chamadas usam `asyncio.to_thread` por serem I/O síncrono (mesmo padrão de `followup_service.py`).
5. Após a ativação, `_dispatch_onboarding_agent` envia uma mensagem de confirmação ao dono informando que a configuração está concluída e o agente já está ativo.
6. Mensagens de tenants em status `onboarding` vindo de phones diferentes de `owner_phone` são ignoradas (descartadas silenciosamente com log de aviso) — o onboarding é exclusivo do dono.
7. `set_tenant_context(tenant_id)` é chamado antes de qualquer query no fluxo de onboarding (NFR3).
8. Falhas em `_dispatch_onboarding_agent` são capturadas com `try/except`, logadas e absorvidas sem propagar — mesmo padrão fire-and-forget de `_dispatch_agent`.
9. Testes unitários em `backend/tests/test_onboarding_dispatch.py` verificam: roteamento para onboarding, roteamento para active, phone diferente de owner_phone descartado, ativação quando `onboarding_complete=True`, ausência do bloco na resposta enviada.
10. CI passa sem regressões nos testes existentes.

## Technical Notes

> O `owner_phone` para validar o AC6 vem de `get_owner_phone(tenant_id)` (Story 5.1). A busca deve ser feita via `asyncio.to_thread` por ser chamada síncrona.
>
> A persistência da mensagem inbound de onboarding já acontece em `_handle_inbound_message` antes do branch (linha existente `_persist_inbound_message`). A mensagem outbound (resposta do agente) deve ser persistida via `persist_outbound_message(tenant_id, phone, content, lead_id=None)` — o `lead_id=None` é válido e já suportado pelo schema.
>
> `_resolve_tenant` já retorna `id, status, name` — o `name` pode ser usado como `tenant_name` no log, mas o agente de onboarding não usa o `tenant_name` no system prompt (a empresa ainda está sendo configurada).
>
> `_dispatch_onboarding_agent` deve ser tolerante a `owner_phone` ausente: se `get_owner_phone` retornar `None`, qualquer phone do tenant em status `onboarding` é tratado como dono (fallback para manter UX funcional antes de `owner_phone` ser preenchido manualmente).
