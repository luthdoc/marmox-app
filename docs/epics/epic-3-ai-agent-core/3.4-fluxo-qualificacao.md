# Story 3.4: Fluxo de Qualificação com System Prompt Dinâmico

**Epic:** 3 — AI Agent Core
**Status:** completed

## User Story

As a marble company owner,
I want the AI agent to collect lead qualification data (name, service, material, urgency, region) naturally in conversation and update the lead record automatically,
so that I receive structured leads without manually reading every message.

## Acceptance Criteria

1. O system prompt injetado no Claude inclui o contexto completo do tenant: nome da empresa, lista de serviços oferecidos, regiões atendidas, horário de funcionamento e mensagem de boas-vindas — todos extraídos do banco via `get_tenant_context(tenant_id) -> dict`.
2. O system prompt instrui o agente a coletar, de forma conversacional e natural: nome do lead, tipo de serviço desejado, material preferido (se aplicável), urgência/prazo e bairro/região (FR3).
3. O system prompt instrui o agente a nunca inventar informações sobre preços ou prazos não cadastrados — se não souber, responder que vai verificar com a equipe (NFR4).
4. Ao final de cada resposta, o agente retorna um bloco JSON estruturado com os dados extraídos até agora (campos: `name`, `service_type`, `material`, `urgency`, `region`, `status`) — o backend faz parse desse bloco e atualiza o lead no banco.
5. O campo `status` do lead transita de `"new"` para `"qualifying"` na primeira extração e para `"qualified"` quando `name`, `service_type` e `region` estiverem preenchidos.
6. `backend/db/leads.py` expõe `update_lead_qualification(lead_id, tenant_id, data: dict)` que faz PATCH nos campos não-None recebidos.
7. Antes de qualquer update em leads, `set_tenant_context(tenant_id)` é chamado (NFR3).
8. O bloco JSON de extração é separado da resposta ao lead — o backend extrai e persiste o JSON, mas envia ao lead apenas o texto da resposta (sem o JSON).
9. `backend/services/agent_service.py` é atualizado para receber `tenant_context: dict` e `lead_data: dict` além dos parâmetros existentes.
10. Testes em `backend/tests/test_qualification_flow.py` cobrem: parse correto do JSON de extração; update de lead com dados parciais; transição de status new→qualifying→qualified; campos ausentes no JSON não apagam dados já existentes no banco.

## Technical Notes

> O agente retorna a resposta ao lead normalmente e, ao final, um bloco delimitado, ex.:
> ```
> [DADOS_LEAD]
> {"name": "João", "service_type": "bancada", "material": null, "urgency": "2 semanas", "region": "Mooca"}
> [/DADOS_LEAD]
> ```
> O backend faz regex/split para separar o texto da resposta do bloco JSON. Se o bloco não aparecer (Claude falhou em incluir), o update é simplesmente ignorado para aquela mensagem.
>
> O system prompt com contexto do tenant é o principal custo de tokens — o prompt caching habilitado em Story 3.1 é essencial aqui. O contexto do tenant deve ser injetado no bloco cacheado.
>
> `get_tenant_context` deve incluir pelo menos: `name`, `services` (list), `regions` (list), `business_hours`, `welcome_message`. Esses campos devem estar na tabela `tenants` — verificar o schema atual e adicionar via migration 002 se ausentes.
>
> Se algum campo de contexto do tenant estiver vazio no banco, o agente deve omiti-lo do prompt (sem "Serviços: []" inútil).
