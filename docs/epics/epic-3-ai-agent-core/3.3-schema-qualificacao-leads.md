# Story 3.3: Schema de Qualificação e Criação de Leads

**Epic:** 3 — AI Agent Core
**Status:** completed

## User Story

As a developer,
I want the database to support full lead qualification data and the agent to create/retrieve a lead record on first contact,
so that qualification data collected in Stories 3.4–3.6 has a place to be stored.

## Acceptance Criteria

1. A migration `backend/db/migrations/002_lead_qualification.sql` adiciona as colunas à tabela `leads`: `service_type TEXT`, `material TEXT`, `urgency TEXT`, `region TEXT`, `scheduled_at TIMESTAMPTZ`, `notes TEXT`.
2. A migration adiciona `owner_phone TEXT` à tabela `tenants` (necessário para notificação ao dono na Story 3.6).
3. A migration adiciona `media_url TEXT` à tabela `messages` (necessário para imagens na Story 3.5).
4. O campo `last_contact_at` da tabela `leads` é atualizado automaticamente via trigger SQL a cada INSERT em `messages` para o `lead_id` correspondente — ou atualizado explicitamente na camada de serviço ao persistir mensagem.
5. `backend/db/leads.py` expõe `get_or_create_lead(tenant_id, phone) -> dict` que: (a) busca lead existente pelo par `(tenant_id, phone)`; (b) se não encontrar, insere com `status = "new"` e retorna o registro criado.
6. Antes de qualquer query em `leads.py`, `set_tenant_context(tenant_id)` é chamado (NFR3).
7. `_dispatch_agent` no `webhook_service.py` chama `get_or_create_lead` e passa `lead_id` para `persist_outbound_message` (Story 3.2).
8. Testes em `backend/tests/test_leads_db.py` cobrem: `get_or_create_lead` com lead inexistente (cria e retorna); lead já existente (retorna sem duplicar); RLS ativo (query com tenant errado retorna vazio).

## Technical Notes

> A migration 002 é aditiva — não altera colunas existentes, apenas adiciona novas (sem risco de regressão no schema existente).
>
> `get_or_create_lead` deve usar upsert ou select-then-insert — cuidado com race condition em ambientes de alta concorrência (para MVP não é bloqueador, mas usar INSERT ... ON CONFLICT DO NOTHING + SELECT subsequente é o padrão seguro).
>
> O `lead_id` retornado por `get_or_create_lead` deve ser passado como `lead_id` ao inserir mensagens inbound e outbound — retroativamente vincular mensagens anteriores está fora do escopo desta story.
