# Story 4.1: Schema e Infraestrutura do Scheduler

**Epic:** 4 — Lead Lifecycle
**Status:** completed

## User Story

As a developer,
I want the database to have the `follow_up_1_sent_at` column and the APScheduler integrated into FastAPI's lifespan,
so that the follow-up job has the schema it needs and runs automatically every hour without manual intervention.

## Acceptance Criteria

1. A migration `004_followup_column.sql` adiciona `follow_up_1_sent_at TIMESTAMPTZ` à tabela `leads` com valor default NULL.
2. A migration é idempotente (`ALTER TABLE ... ADD COLUMN IF NOT EXISTS`).
3. A dependência `apscheduler>=3.10` está presente em `backend/requirements.txt`.
4. O `main.py` define um `lifespan` context manager que inicia o `BackgroundScheduler` no startup e chama `scheduler.shutdown()` no shutdown.
5. O scheduler registra o job `run_followup_job` com trigger `interval` de 60 minutos, identificado com `id="followup_job"`.
6. A função `run_followup_job` é importada de `services.followup_service` (pode ser um stub nesta story).
7. Testes unitários verificam que o scheduler inicia sem erros e que o job `"followup_job"` está presente na lista de jobs após o startup.
8. Testes unitários verificam que `scheduler.shutdown()` é chamado corretamente quando o lifespan encerra.

## Technical Notes

> O `main.py` atual não possui lifespan. O `BackgroundScheduler` do APScheduler roda em thread separada — não bloqueia o event loop do FastAPI. O lifespan deve ser registrado via `app = FastAPI(lifespan=lifespan_fn)`.
>
> O stub de `run_followup_job` em `followup_service.py` deve apenas fazer `pass` (ou logar) para que a importação não falhe — a implementação real vem na Story 4.3.
>
> A migration deve ser executada no Supabase SQL Editor. O arquivo deve incluir comentário explicitando que é pré-requisito para a Story 4.2.
