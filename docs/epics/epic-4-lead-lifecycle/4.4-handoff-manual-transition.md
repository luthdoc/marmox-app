# Story 4.4: Endpoint de Handoff Manual

**Epic:** 4 — Lead Lifecycle
**Status:** completed

## User Story

As a marble company owner (via dashboard),
I want to manually transition a lead to `handoff` status through an API endpoint,
so that I can signal that I am personally taking over the conversation and the lead stops receiving automated follow-ups.

## Acceptance Criteria

1. `PATCH /leads/{lead_id}/handoff` está disponível e retorna 200 com o lead atualizado quando a transição é válida.
2. O endpoint aceita `X-Tenant-ID` como header obrigatório para identificar o tenant autenticado (mesmo padrão de header do projeto).
3. O endpoint retorna 400 se o header `X-Tenant-ID` estiver ausente ou em formato inválido (não UUID).
4. O endpoint retorna 404 se o lead não existir ou não pertencer ao tenant informado no header.
5. O endpoint retorna 409 se o lead já estiver com `status = 'cold'` — regressão de status é proibida.
6. Leads em qualquer outro status (`new`, `qualifying`, `qualified`, `scheduled`, `handoff`) são transicionados para `handoff` com sucesso (idempotente para o próprio `handoff`).
7. Antes de qualquer query ao banco, `set_tenant_context(tenant_id)` é chamado (NFR3) — garante que o RLS impede acesso a leads de outros tenants.
8. Nenhum dado de outro tenant é exposto: a busca do lead usa `tenant_id` do header como filtro explícito além do RLS.
9. O router está em `backend/routers/leads.py` e é registrado em `main.py` via `app.include_router(leads_router)`.
10. Testes cobrem: 200 com transição correta a partir de cada status elegível; 404 para lead inexistente; 404 para lead de outro tenant; 409 para lead cold; 400 para header ausente.

## Technical Notes

> O padrão de autenticação atual do projeto usa `X-Zapi-Token` no webhook — não há middleware de auth por tenant para rotas REST ainda. Para esta story, usar `X-Tenant-ID` como header simples contendo o UUID do tenant. O dashboard (Epic 6) passará este header. Não implementar JWT ou Supabase Auth nesta story.
>
> O response body do 200 deve incluir ao menos: `id`, `tenant_id`, `phone`, `name`, `status`, `last_contact_at`.
>
> A validação de UUID do `tenant_id` pode ser feita com `uuid.UUID(tenant_id_str)` dentro de um try/except — retornar 400 em caso de ValueError.
>
> O endpoint não envia notificação ao dono nem ao lead — apenas atualiza o status. Notificações de handoff, se necessárias, serão responsabilidade da Epic 6 ou de uma story futura.
