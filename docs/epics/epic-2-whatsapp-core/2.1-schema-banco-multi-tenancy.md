# Story 2.1: Schema do Banco e Multi-tenancy Base

**Epic:** 2 — WhatsApp Core
**Status:** completed

## User Story

As a developer,
I want the Supabase database schema with `tenants`, `leads`, and `messages` tables with RLS enabled and a working Python client,
so that every subsequent story can persist and query data with tenant isolation guaranteed at the database level.

## Acceptance Criteria

1. A migration SQL (`backend/db/migrations/001_initial_schema.sql`) cria as tabelas `tenants`, `leads` e `messages`, cada uma com coluna `tenant_id UUID NOT NULL` e chave estrangeira para `tenants.id`.
2. Row Level Security está habilitado em `leads` e `messages` com política `USING (tenant_id = current_setting('app.tenant_id')::uuid)` — uma query executada com `tenant_id` de outro tenant não retorna nem altera registros do tenant correto.
3. O cliente Supabase (`backend/db/client.py`) é inicializado com `SUPABASE_URL` e `SUPABASE_SERVICE_KEY` via `core/config.py` e expõe uma função `get_client()` que retorna a instância singleton.
4. A função `set_tenant_context(tenant_id: str)` define `app.tenant_id` na sessão Postgres antes de qualquer query — testada com uma query direta que confirma isolamento.
5. O módulo `backend/db/__init__.py` exporta `get_client` e `set_tenant_context`.
6. Um teste unitário em `backend/tests/test_db_schema.py` verifica que `get_client()` retorna uma instância válida e que `set_tenant_context` não lança exceção com UUID válido.

## Technical Notes

> Usar `supabase-py` (client oficial). O service key bypassa RLS por padrão — para enforcement de RLS, as queries devem setar `app.tenant_id` via `rpc('set_config', ...)` ou via SQL raw antes de cada operação. Em produção, o fluxo será: receber tenant_id da requisição → chamar `set_tenant_context` → executar queries.
>
> Schema mínimo:
> - `tenants`: id, name, status (onboarding|active), zapi_instance_id, zapi_token, created_at
> - `leads`: id, tenant_id, phone, name, status (new|qualifying|qualified|scheduled|handoff|cold), created_at, last_contact_at
> - `messages`: id, tenant_id, lead_id, direction (inbound|outbound), content, created_at
>
> Índices obrigatórios: `leads(tenant_id)`, `messages(tenant_id, lead_id)`.
