# Story 2.4: Echo Bot de Ponta a Ponta

**Epic:** 2 — WhatsApp Core
**Status:** completed

## User Story

As a marble company owner (tenant),
I want my WhatsApp number to echo back any text message I send to it,
so that the end-to-end flow from webhook reception to message delivery is validated before the AI agent is introduced.

## Acceptance Criteria

1. Ao receber uma mensagem de texto via `POST /webhook/whatsapp` de um tenant ativo (`status = "active"`), o sistema responde automaticamente ao mesmo número com `"Recebi: [texto original]"`.
2. O echo só ocorre para tenants com `status = "active"` — mensagens de tenants em `status = "onboarding"` são persistidas mas não geram resposta automática.
3. A resposta de echo é enviada via `send_message` do serviço Z-API implementado na Story 2.3 — o webhook retorna HTTP 200 antes de aguardar o envio (fire-and-forget com `asyncio.create_task`).
4. Se o envio do echo falhar, o erro é logado mas não afeta a resposta HTTP do webhook (que já retornou 200).
5. O fluxo completo é coberto por um teste de integração em `backend/tests/test_echo_bot.py`: mensagem recebida de tenant ativo → `send_message` chamado com o texto correto; mensagem de tenant em onboarding → `send_message` não chamado.
6. Um teste manual documentado em `docs/epics/epic-2-whatsapp-core/manual-test-echo.md` descreve os passos para validar o fluxo real com Z-API configurado (seed de tenant no banco + envio de mensagem pelo WhatsApp + verificação do echo recebido).

## Technical Notes

> Esta story integra as Stories 2.2 e 2.3. A lógica de dispatch do echo deve ficar em `backend/services/webhook_service.py` — o service já processa o inbound, agora também dispara o echo para tenants ativos.
>
> Fire-and-forget com `asyncio.create_task` garante que o webhook retorne imediatamente (requisito NFR1 de < 5s de resposta). O AI agent da Epic 3 substituirá essa lógica de echo pelo orquestrador real — a arquitetura deve tornar essa substituição trivial (trocar o callback, não refatorar o webhook).
>
> Para o teste manual, é necessário um tenant seed: inserir uma linha em `tenants` com `status = "active"`, `zapi_instance_id` e `zapi_token` reais. Documentar o SQL de seed no arquivo de teste manual.
