# Story 2.3: Serviço de Envio de Mensagens via Z-API

**Epic:** 2 — WhatsApp Core
**Status:** completed

## User Story

As a developer building the AI agent,
I want a `send_message(tenant_id, phone, text)` function that sends WhatsApp messages via Z-API,
so that any part of the system can send messages without knowing Z-API's HTTP details.

## Acceptance Criteria

1. A função `send_message(tenant_id: str, phone: str, text: str) -> bool` em `backend/services/zapi_client.py` envia uma requisição POST para o endpoint Z-API correto e retorna `True` em sucesso (HTTP 2xx).
2. As credenciais Z-API (`instance_id` e `token`) são lidas da tabela `tenants` pelo `tenant_id` — cada tenant pode ter instância própria.
3. Em falha de rede ou resposta não-2xx, a função faz até 3 tentativas com backoff exponencial (1s, 2s, 4s) antes de retornar `False`.
4. Toda mensagem enviada gera log INFO estruturado com `tenant_id`, `phone`, `message_length`, `attempt_number` e `success`.
5. Em caso de falha após todas as tentativas, gera log ERROR com `tenant_id`, `phone`, `error` e retorna `False` sem lançar exceção.
6. A mensagem enviada é persistida em `messages` com `direction = "outbound"` após envio bem-sucedido.
7. Um teste em `backend/tests/test_zapi_client.py` cobre: envio com sucesso (mock HTTP 200), falha na 1ª tentativa e sucesso na 2ª (mock), falha em todas as tentativas (mock HTTP 500 repetido → retorna False).

## Technical Notes

> Endpoint Z-API para envio de texto:
> `POST https://api.z-api.io/instances/{instance_id}/token/{token}/send-text`
> Body: `{"phone": "5511999999999", "message": "texto"}`
>
> Usar `httpx.AsyncClient` para as chamadas HTTP (já é dependência indireta via `supabase-py` em alguns setups; adicionar explicitamente ao `requirements.txt`). O backoff pode ser implementado com `asyncio.sleep` — não adicionar dependência de biblioteca de retry no MVP.
>
> O lookup de credenciais do tenant deve ser cacheado em memória por curto período (30s) para evitar query ao banco a cada mensagem enviada.
